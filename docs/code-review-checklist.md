# Pulse 代码质量审查清单

> 每完成一个功能模块后，将本文档连同代码一起提交给大模型进行审查。

---

## 核心原则：第一性原理

写每一行代码前问自己：**这个问题的本质是什么？最简单正确的解法是什么？**

不要因为"别人这么写"或"先跑起来再说"就绕过思考。

---

## 两种屎山代码（零容忍）

屎山的本质是**逃避问题**：用防御代码盖住 bug,或在下游打补丁绕过根本问题。

### 类型 A：防御式逃避（吞异常 + 兜底回退）

检查项：
- [ ] `except Exception` / 裸 `except` → 是否可缩小到具体异常类型？
- [ ] `except` 分支是否记录了 `logger.exception()`？还是悄悄吞了？
- [ ] `except: return None / [] / {}` → 错误是否应该让调用方感知,而不是伪造"空结果"？
- [ ] `a or b or c or default` 层层兜底 → 能否让上游保证 `a` 一定有效值?
- [ ] 这个异常是**该处理**(网络抖动、用户输入)还是**该暴露**(配置缺失、代码 bug)?后者必须 fail-fast。

### 类型 B：补丁式兼容(不修源头,下游堆补丁)

检查项:
- [ ] 代码里有 `# HACK` / `# FIXME` / `# workaround` / `# TODO: 临时` → 逐条追溯根因
- [ ] `isinstance` 分支 / `get("a") or get("A") or get("a_b")` → 上游为什么会返回多种形态?能否修上游或收敛到单一 adapter?
- [ ] 同一份"脏数据"是否在多处被反复 normalize? → 在入口洗一次
- [ ] 硬编码路径 / URL / 魔法数字 → 是否绕过了 `Settings`?
- [ ] 这个补丁的生命周期是什么?若"永远存在",它就是债务,必须立项重构

### 类型 C:过度设计(次要,但也要清理)

检查项:
- [ ] 只有一个实现的抽象类/接口(除非是明确扩展点)
- [ ] "将来可能用到"的参数/配置项/泛型 → 删掉
- [ ] 可以用简单函数替代的类 → 降级

## 测试宪法(零容忍虚假测试)

> 测试也是代码,一样会屎山化。虚假测试比没测试更坏——它给你"已验证"的幻觉,同时把 CI 时间、token、心智成本一起烧掉。

### 测试是什么,不是什么

测试的本质是**给"这段代码在真实场景下能 work"留下可复现的证据**,面向两类读者:① 改坏这段代码的未来自己,② 审查这次改动的人。

测试**不是**:
- "我改对了"的**心理按摩** → 表演式反复跑
- Coverage 数字的燃料 → 为刷分而加 test
- 同一份逻辑的**影子复制品** → 源码改一行,测试跟着改一行(测的是实现不是行为)

### 零容忍的虚假测试(写 test 前自查)

- [ ] **同源复读**:test 断言的是**你自己刚写的常量/默认值/格式化字符串**(`assert _MAX == 8`,`assert render(x) == "flag=true"`) → 删,它不拦任何 regression,只拦你"改主意"
- [ ] **形态造假**:test payload 是手搓的 dict,但真实上游返回的 shape 不同(多/少字段、嵌套不同、int 实际是 str、带了 PUA 码点) → 用**真实 trace / provider 响应**切片存成 fixture,不要脑补
- [ ] **过度 mock**:被测函数的所有依赖都被 mock,断言退化成"mock 被调了 N 次 / 传参是 X" → 测的是调用图不是行为;提升到集成层或拆开只测纯函数
- [ ] **测第三方**:断言 stdlib / pydantic / pytest / fastapi / sqlalchemy 的已知行为 → 不是你的职责
- [ ] **per-field unit**:为一个 dataclass 的每个 getter 单独写 test → 聚合成一个"整体契约"test
- [ ] **绿色但无声**:一个 test 永远绿,删了也没人发现异常 → 它没有区分力,删

### 测试分层(投入产出比从高到低)

1. **契约 / 集成测试** — 真实上下游拼起来,只在**环境边界**(网络 / 外部服务 / 时间 / 随机)mock。一个 test 覆盖一条"真实调用链",回报最高
2. **单元测试** — 纯函数 + 复杂分支逻辑,AAA(Arrange-Act-Assert)结构,一个 test 一个概念
3. **真实 trace 回放** — 合入前 / bug 复现后,拿**生产 trace**(存到 `logs/` 或 audit jsonl)做一次 replay,覆盖上面两层加起来仍会漏的接缝。**这才是"不假通过"的最后一道闸**
4. 不做:UI 截图级 e2e、只测 getter/setter 的 per-field unit、为了覆盖率补的边角 test

注:业界共识可参考 Kent Beck "Test desired behavior, not implementation"、Kent C. Dodds "Testing Trophy"(集成 > 单元金字塔)、Martin Fowler "Write tests. Not too many. Mostly integration."

### 写 test 的纪律

- [ ] 断言**行为**(输入 → 输出 / 副作用 / 可观察状态),不是**实现**(私有方法、调用次数、内部字段值)——后者是反 refactor 的负资产
- [ ] 一个 test 失败能**独立定位**根因,不需要翻 5 层 mock 才看懂
- [ ] 一条改动需要改 5 个 test 才能过 → 测的是实现不是契约,重写
- [ ] 被测代码删除 / 改 signature → 对应 test **同步删**,不要留僵尸

### 跑 test 的纪律(反表演)

- [ ] **一批改动做完再跑一次**,禁止"改一行跑一遍" — 后者是心理按摩,浪费资源
- [ ] 只跑与改动**语义相关**的 suite;全量回归只在合并前 / CI 跑一次
- [ ] 全绿 ≠ 功能对 — 合入前必须**额外**对**真实 trace / 真实用户动作**做一次端到端验收,否则不算通过
- [ ] pre-existing fail 与本次改动 **diff-aware** 隔离判定(看"这条 fail 是我引入的吗",不看总数)
- [ ] 同一组断言第二次就过了,却仍反复跑 → 不是严谨,是逃避"改下一步" 的焦虑,立即停

## 逐项审查清单

### 1. 配置与依赖
- [ ] 配置通过 `Settings` 统一管理，不在业务代码里 `os.getenv()`
- [ ] 依赖通过构造函数注入，不在 `__init__` 里读全局状态
- [ ] `.env.example` 与 `.env` 字段同步

### 2. 错误处理与日志
- [ ] 不吞异常（空 `except: pass`）
- [ ] 关键路径有日志（启动、连接、收发消息），异常路径有 WARNING/ERROR
- [ ] 日志用 `logging.getLogger(__name__)`，不用 `print`
- [ ] 日志不打印完整密钥（最多前 10 位）

### 3. 异步代码
- [ ] 协程没有遗漏 `await`
- [ ] 后台任务保存了引用，shutdown 时正确取消

### 4. 清洁度
- [ ] 没有未使用的 import
- [ ] 没有注释掉的代码块
- [ ] 没有临时调试代码（print、硬编码测试值）
