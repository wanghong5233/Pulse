# OfferPilot V2 架构调研与决策文档

> 调研日期：2026-03-14
> 调研范围：OpenClaw 社区生态、业界 AI Agent 框架、社区同类项目、LangGraph 最新范式
> 目标：确定 OfferPilot 从"求职执行 Agent"演进到"Jarvis 级通用个人智能体"的最优架构

---

## 〇、需求分析

### 0.1 产品定位

OfferPilot 的终极形态是一个**通用个人智能体（Personal Agent）**——类似钢铁侠的 Jarvis。当前阶段的任务是找工作，但这只是第一个"技能包"。底层架构必须是**领域无关、能力可插拔、自我进化**的通用 Agent 平台。

**两类能力并存**（这是与"纯 Pipeline 自动化平台"的根本区别）：

| 能力类型 | 核心模式 | 典型场景 | 触发方式 |
|---------|---------|---------|---------|
| **Pipeline（重型自动化）** | 定时/手动触发 → 多源采集 → LLM 处理 → 执行/通知 | 投递、面经采集、技术雷达、深度调研 | Scheduler 定时 / 用户指令 |
| **Tool Call（轻量级交互）** | 用户提问 → Brain 推理 → 调用工具 → 返回结果 | 设闹钟、查天气、播放音乐、创建日历事件、查航班 | 实时对话 |

> **关键洞察**：如果只有 Pipeline，OfferPilot 永远是"自动化脚本集合"。只有同时具备 Brain（推理循环）+ Tool Call（轻量级工具调用），才是真正的"智能助手"。

**场景矩阵**（说明通用性）：

| 场景 | 类型 | 触发 | 数据源/工具 | 处理 | 输出 |
|------|------|------|-----------|------|------|
| 求职：自动投递 | Pipeline | 定时 | Patchright → BOSS | LLM 匹配评分 | 打招呼 → 飞书通知 |
| 求职：面经情报 | Pipeline | 每日 08:00 | 当前实现：Web Search；目标态：Patchright → 牛客/脉脉 | LLM 提取考点 | 飞书日报推送 |
| 技术雷达 | Pipeline | 每日 09:00 | 当前实现：Web Search；目标态：RSS / GitHub / 公众号 | LLM 筛选摘要 | 飞书日报推送 |
| 深度调研 | Pipeline | 手动 | Web 搜索 + 爬取 | LLM 汇总分析 | 结构化报告 |
| 设置闹钟 | Tool Call | 对话 | `set_alarm(time, label)` | — | 确认回复 |
| 查询天气 | Tool Call | 对话 | `get_weather(city)` API | — | 天气信息 |
| 安排行程 | Tool Call | 对话 | `calendar.create_event(...)` | — | 日历已创建 |
| 播放音乐 | Tool Call | 对话 | MCP → Spotify Server | — | 播放中 |
| 组合任务 | Brain 多步 | 对话 | 查天气 → 查航班 → 建日程 | Brain 推理串联 | 出行方案 |
| 旅游攻略 | Pipeline | 手动 | 旅游网站 + 点评 | LLM 筛选推荐 | 攻略文档 |
| 财务管理 | Pipeline | 每周 | 银行/券商 API | LLM 分析趋势 | 周报推送 |
| 自扩展 | Meta | 对话 | "我需要监控 BTC 价格" | LLM 生成代码 | 新 Tool/Module 热加载 |

**架构要求**：
- 当前求职功能定义的架构底座，未来不需要重建
- 新增 Pipeline 场景（如旅游攻略）→ 新建一个 Module 目录
- 新增 Tool Call 场景（如查天气）→ 新建一个 Tool 函数（或接入 MCP Server）
- 新增组合能力 → Brain 自动编排已有 Tool + Module，无需新代码

### 0.1.1 当前阶段演进

| 阶段 | 定位 | 核心架构升级 | 核心用户价值 |
|------|------|------------|------------|
| Phase 1（已完成） | 求职执行 | 单体脚本 | BOSS 自动投递 + 自动回复 + 邮件追踪 |
| Phase 2（本次目标） | 求职 + 情报 | **Module + Capability + Router** | 面经采集 + 技术雷达 + 语义检索 |
| Phase 3（近期） | 通用助手 | **Router → Brain + Tool Registry + MCP Client + 基础记忆（Core/Recall Memory）** | 轻量级工具调用 + 多步推理 + 外部生态 + 跨会话记忆 |
| Phase 4（远期） | 自进化助手 | **Skill Generator（LLM 生成代码热加载）** | 用户自然语言创建新能力 |
| Phase 5（长期） | 自主进化 | **Evolution Engine（人格进化 + 偏好学习 + 时序事实图 + 可选 DPO 微调）** | 反思式自主进化、越用越聪明、个性化适应 |

> Phase 2 架构是 Phase 3/4/5 的**严格子集**。Router 是 Brain 的退化形式（单步推理 = ReAct 循环只执行一轮）。Module 不变。Capability Layer 只做增量新增。每次升级都是**增量式**的，不需要重写。

### 0.2 Phase 1 功能清单（已完成，作为基线）

**实际在生产运行的功能（Guard 调度 + API 调用）**：

| 功能 | 触发方式 | 数据源 | 执行方式 | 输出 |
|------|---------|--------|---------|------|
| **岗位搜索 + JD 分析** | Guard 定时 / 用户手动 | BOSS 直聘搜索页 | Patchright 抓取 + LLM 解析评分 | 岗位列表入库 + 向量化 |
| **主动打招呼** | Guard 高峰期定时 | BOSS 直聘详情页 | Patchright 点击"立即沟通" + LLM 精筛 | 打招呼 + 审计记录 |
| **聊天智能回复** | Guard 3min 巡检 | BOSS 直聘聊天页 | Patchright 拉消息 + LLM 意图识别 + 四层门控 + 自动回复 | 回复/通知用户 |
| **邮件追踪** | EmailHeartbeat 独立线程 | IMAP 邮箱 | 拉取 + LLM 分类 + 日程提取 | 状态更新 + 飞书通知 |

**代码存在但 V2 不保留的功能**（砍掉）：

| 功能 | 状态 | V2 处理 |
|------|------|--------|
| 材料生成（material_workflow） | 代码完整，V2 不需要 | 移除 |
| 面试准备（interview_prep_service） | 代码完整，V2 不需要 | 移除 |
| 公司情报（company_intel_service） | 代码完整，从未实际使用 | 移除 |
| 表单自动填充（form_fill_workflow） | 代码完整，V2 不需要 | 移除 |

**Phase 1 核心技术约束（不可改变）**：
- Patchright（Python）做 BOSS 直聘浏览器自动化，binary-level 反检测
- 浏览器会话必须在单一 Python 进程内共享（persistent_context 保持登录态）
- 所有 LLM 调用使用 Pydantic Structured Output（类型安全 + 校验重试）
- PostgreSQL 存储业务数据，ChromaDB 存储向量
- ProductionGuard 做时段感知调度 + 资源治理

> 说明：OpenClaw Gateway 是 V1 的接入方式，不是不可替代的技术硬约束。V2 将其降级为可选兼容层（或完全替换）。

### 0.3 Phase 2 新增需求（本次开发目标）

#### 需求 F1：面经情报流（Interview Intelligence）

| 项 | 内容 |
|----|------|
| **用户故事** | 作为求职者，我希望每天自动收到目标公司的面经热点，不用自己去各平台搜 |
| **数据源** | 当前实现：通用 Web Search；目标态：牛客网、脉脉、小红书（反爬站点，需要 Patchright） |
| **采集频率** | 每天 1 次（08:00），可手动触发 |
| **处理逻辑** | URL 去重 → 内容相似度去重 → LLM 提取（轮次/考察点/编程题/难度）→ 按公司聚合 |
| **输出** | 每日面经简报 → 飞书卡片推送（标题 + 高频考点 + 原文链接） |
| **存储** | 结构化面经条目入 PostgreSQL + 向量化入 ChromaDB（支持语义检索） |
| **信息治理** | 相关度评分过滤（LLM 评分 < 阈值不进日报）、去重漏斗、每日限 5-8 条 |
| **配置** | 目标公司列表、目标岗位关键词、采集源开关、相关度阈值 |

#### 需求 F2：技术雷达流（Tech Radar）

| 项 | 内容 |
|----|------|
| **用户故事** | 作为开发者，我希望每天收到我关注领域的技术动态精选，不用逐个翻公众号和博客 |
| **数据源** | RSS 源（博客/36Kr/量子位/机器之心）、GitHub Trending、微信公众号历史文章（需要 Patchright） |
| **采集频率** | 每天 1 次（09:00），可手动触发 |
| **处理逻辑** | 拉取更新 → LLM 相关度评分（0-100）→ 阈值过滤 → LLM 3 句话摘要 + 标签 → 聚合日报 |
| **输出** | 每日技术简报 → 飞书卡片推送（标题 + 摘要 + 标签 + 原文链接） |
| **存储** | 情报条目入 PostgreSQL + 向量化入 ChromaDB |
| **信息治理** | 同 F1：相关度门控 + 去重 + 摘要压缩 + 每日限 5-8 条 |
| **配置** | 关注源列表（`intel_sources.yaml`）、关注关键词、相关度阈值 |

#### 需求 F3：情报语义检索

| 项 | 内容 |
|----|------|
| **用户故事** | 作为求职者，我想问"这周有没有关于 MCP 的新动态"并得到精准答案 |
| **实现** | 面经 + 技术情报向量化入 ChromaDB → 用户通过飞书/Web 提问 → 向量检索 + LLM 回答 |
| **依赖** | F1 和 F2 的数据入库 |

### 0.4 Phase 3-5 需求（近期 + 远期 + 长期，架构预留 + 设计完成）

#### 需求 F4：内容发布

| 项 | 内容 |
|----|------|
| **流程** | 选取日报内容 → LLM 改写为推文风格 → 人工审批 → 多平台发布（微信公众号/知乎/即刻） |
| **关键点** | 必须有人工审批环节（interrupt）、多平台 API 适配、发布结果追踪 |

#### 需求 F5：用户反馈闭环

| 项 | 内容 |
|----|------|
| **流程** | 用户对日报内容标记"有用/无用" → 反馈信号调整采集权重和相关度模型 |
| **关键点** | 需要反馈存储 + 权重调整机制 |

#### 需求 F6：Brain 推理引擎（Router → Brain 升级）

| 项 | 内容 |
|----|------|
| **用户故事** | 作为用户，我希望和 OfferPilot 进行多轮自然对话，它能理解复杂指令并自动编排多个工具完成任务 |
| **核心变化** | Phase 2 的 Router（一次 LLM 调用 → 分类到一个 Module）升级为 Brain（ReAct 推理循环：Think → Act → Observe → Think → ...） |
| **推理循环** | ① 接收用户消息 + 加载记忆上下文 → ② 看到所有可用能力（Tool + Module + MCP Tool）→ ③ LLM 决定下一步（调用工具 or 直接回复）→ ④ 执行工具并获取结果 → ⑤ 结果追加到上下文 → ⑥ 循环直到 LLM 产生最终文本回复 |
| **终止条件** | LLM 返回纯文本（无 tool_call）→ 循环结束；安全阀：maxToolCalls = 20 / maxConsecutiveErrors = 3 |
| **与 Router 兼容** | Brain 是 Router 的超集。Phase 2 的 Router 等价于 "Brain 只执行一轮循环"。升级不改 Module 接口 |
| **参考实现** | OpenClaw Brain 6步循环、Pincer ReAct Loop、Orchestral AI 同步推理 |
| **记忆集成** | Brain 每轮推理前加载相关记忆上下文（Core Memory + 相关 Recall Memory），推理后自主决定是否更新记忆 |

#### 需求 F7：Tool Registry（轻量级工具系统）

| 项 | 内容 |
|----|------|
| **用户故事** | 作为用户，我希望对 OfferPilot 说"帮我设个明天早上7点的闹钟"，它直接完成，不需要启动任何重型 Pipeline |
| **与 Module 的区别** | Module = 重型 Pipeline（有调度、有生命周期、有 DB）；Tool = 轻量级单函数（无调度、无状态、毫秒级执行） |
| **工具格式** | 兼容 MCP 标准：函数名 + 描述 + JSON Schema 参数。使用 FastMCP `@mcp.tool` 装饰器自动从 type hints 生成 schema |
| **内置工具示例** | `set_alarm`, `set_timer`, `get_weather`, `search_web`, `read_url`, `run_python`, `create_note`, `list_tasks` |
| **发现机制** | 启动时扫描 `tools/` 目录，自动注册到 Brain 的能力菜单。与 Module 的 capability 声明合并为统一的 tool list |
| **MCP 兼容** | OfferPilot 的 Tool 同时作为 MCP Server 对外暴露，可被 Cursor/Claude Desktop 等任何 MCP Client 调用 |

#### 需求 F8：MCP Client（外部工具生态接入）

| 项 | 内容 |
|----|------|
| **用户故事** | 作为用户，我希望 OfferPilot 能调用我的 Google Calendar、Spotify、智能家居等外部服务，而不需要为每个服务写代码 |
| **实现方式** | OfferPilot 作为 MCP Client，连接外部 MCP Server；运行时自动发现外部工具并加入 Brain 的能力菜单 |
| **生态规模** | 2026 年已有 10,000+ 公开 MCP Server（文件系统、数据库、日历、搜索、音乐、智能家居...）|
| **配置** | `mcp_servers.yaml` 列出要连接的 MCP Server 地址，支持 stdio/HTTP+SSE/Streamable HTTP 三种传输协议 |
| **安全** | Policy Engine 门控：外部 MCP 工具首次调用需要用户确认；可配置白名单和黑名单 |

#### 需求 F9：Skill Generator（自然语言创建新能力）—— Phase 4

| 项 | 内容 |
|----|------|
| **用户故事** | 作为用户，我说"我需要一个每天监控比特币价格的功能"，OfferPilot 自动生成代码、测试、部署，立即可用 |
| **实现方式** | Meta-Tool：LLM 分析需求 → 决定生成 Tool 还是 Module → 生成代码 → AST 安全扫描 → 沙箱测试 → 热加载到运行时 |
| **安全五层防线** | ① AST 语法检查 → ② import 白名单（只允许 httpx/json/datetime 等安全模块）→ ③ 模式检测（阻止混淆/eval/exec）→ ④ subprocess 沙箱执行（5s 超时 / 256MB 内存限制）→ ⑤ Policy Engine 用户确认激活 |
| **参考方案** | OpenClaw create-skill（10秒部署新技能）、AutoAgent（零代码Agent生成）、Pincer Skills 沙箱 |
| **热加载** | 使用 `importlib.reload()` + 文件监听（watchfiles），新生成的 Tool/Module 无需重启即可使用 |

#### 需求 F10：基础记忆系统（Core + Recall Memory）—— Phase 3

| 项 | 内容 |
|----|------|
| **用户故事** | 作为用户，我希望 OfferPilot 记住我的姓名、偏好、历史对话，不需要每次重新告知 |
| **Core Memory** | 每轮推理加载的工作记忆，包含 4 个 Block：SOUL（人格）、USER（用户画像）、PREFS（偏好规则）、CONTEXT（当前任务） |
| **Recall Memory** | 全量对话历史 + 工具调用记录，按时间和语义双索引，推理前自动检索相关片段注入上下文 |
| **记忆工具** | Brain 通过 `memory_read/memory_update/memory_search` 工具自主管理记忆，与其他 Tool 同级 |
| **容量管理** | Core Memory < 50K 字符；超限时 Brain 自动精简（摘要化后移入 Recall） |
| **存储** | Core Memory → PostgreSQL JSONB；Recall → PostgreSQL + 向量索引 |
| **详细设计** | 见第十节 §10.3 |

#### 需求 F11：人格进化与偏好学习 —— Phase 5

| 项 | 内容 |
|----|------|
| **用户故事** | 作为用户，我希望 OfferPilot 能从我的纠正和反馈中学习，越用越懂我 |
| **人格进化** | SOUL Block 中的 `[MUTABLE]` 信念可通过反思 Pipeline 进化；`[CORE]` 信念不可修改（安全底线） |
| **偏好学习（Track A）** | 纠正检测 → 规则提取 → PREFS Block 更新。无需 GPU，覆盖 80% 个性化效果 |
| **偏好学习（Track B，可选）** | 收集 DPO 训练对 {query, chosen, rejected}。需 GPU 时可接入微调管线 |
| **治理门控** | 三级治理：Autonomous（自动应用低风险偏好）/ Supervised（用户审核）/ Gated（显式批准） |
| **时序事实图** | PostgreSQL JSONB 轻量时序图谱，事实三元组 + 时间戳 + 有效窗口，支持事实更迭追踪 |
| **详细设计** | 见第十节 §10.3.5-10.3.6 |

### 0.5 非功能需求

| 类别 | 需求 | 优先级 | 阶段 |
|------|------|--------|------|
| **通用可扩展性** | 底层架构一次定义，新增 Pipeline 场景加 Module 目录，新增工具场景加 Tool 函数或接入 MCP Server，均不改框架代码 | **P0** | Phase 2 |
| **领域无关性** | 核心框架（调度、推理、LLM、通知、存储）不包含任何求职相关的业务逻辑，求职只是一个 Module | **P0** | Phase 2 |
| **功能独立性** | 每个 Module/Tool 自包含，模块之间低耦合（不共享代码，仅通过存储约定共享数据），可独立开发/测试/启停 | P0 | Phase 2 |
| **浏览器按需** | 需要浏览器的功能共享实例；不需要浏览器的功能（RSS/API/Tool Call）不应有浏览器依赖 | P0 | Phase 2 |
| **调度灵活性** | 不同模块有不同的调度策略（高峰加密 vs 每日一次 vs 手动触发），模块自声明调度需求 | P0 | Phase 2 |
| **安全** | 任何自动执行操作都有门控/审核机制 | P0 | Phase 2 |
| **MCP 兼容性** | 内置工具以 MCP Server 标准对外暴露；可作为 MCP Client 接入外部工具生态（10,000+ servers） | **P0** | Phase 3 |
| **多步推理** | Brain 支持 ReAct 推理循环，能将多个 Tool/Module 调用串联完成复杂任务（如"查天气→查航班→建日程"） | **P0** | Phase 3 |
| **自扩展能力** | 用户通过自然语言描述需求，系统自动生成新 Tool/Module 代码并热加载 | P1 | Phase 4 |
| **可观测性** | 所有 Pipeline/Brain 推理链有日志 + 事件流 + 审计记录 | P1 | Phase 2 简版 / Phase 3 全栈 |
| **通知** | 飞书分级推送（日报/告警/摘要）、SSE 前端实时展示 | P1 | Phase 2 |
| **容错** | 单条失败不阻断批量，LLM 主备降级，浏览器崩溃自动恢复，Brain 推理循环安全阀（maxToolCalls） | P1 | Phase 2 |
| **部署简单** | 跨平台单机部署（Windows / Linux / macOS），单进程，一键拉起；可选 Docker Compose 全栈启动 | P1 | Phase 2 |
| **成本可控** | 每日 LLM 消费硬上限 + 实时追踪 + 预算耗尽自动降级模型（参考 Pincer 的 cost control） | P1 | Phase 3 |
| **跨会话记忆** | Brain 具备长期记忆能力，能在不同会话间记住用户偏好、历史交互、关键事实 | **P0** | Phase 3 |
| **记忆自主管理** | Agent 能自主决定哪些信息值得记忆、何时遗忘、如何压缩，不依赖用户手动管理 | P1 | Phase 3 |
| **人格一致性** | Agent 具备可配置的人格（语气、风格、价值观），跨会话保持一致 | P1 | Phase 3 |
| **偏好学习** | Agent 从用户纠正、反馈、行为模式中学习，逐步适应用户的个性化需求 | P1 | Phase 5 |
| **数据可追溯** | 所有记忆变更、人格进化、偏好调整均有审计日志，用户可查看和回滚 | P1 | Phase 5 |
| **配置驱动** | 所有阈值/频率/源列表/开关通过配置文件控制 | P1 | Phase 2 |
| **可分发性** | 模板/模块可分享，可被社区一键安装和复用 | P1 | Phase 3 |
| **路由可预测性** | 消息路由由主机策略控制，模型不直接决定发送渠道，避免误发 | P0 | Phase 2 |

### 0.6 技术约束分析

区分**真约束**（不可协商）和**当前选择**（可以换但没必要换）：

**真约束（不可协商）**：

| 约束 | 为什么不可协商 |
|------|--------------|
| **Python 运行时** | Patchright 是 Python 库，BOSS 直聘自动化必须用它 |
| **Patchright 用于 BOSS 直聘** | BOSS 有反自动化检测，binary-level CDP 反检测是唯一可靠方案。标准 Playwright 会被封 |
| **BOSS 浏览器会话进程内共享** | persistent_context 保持 BOSS 登录态，Patchright page 对象不可序列化，必须同一进程内操作 |

**当前选择（强偏好但非约束）**：

| 选择 | 当前状态 | 能不能换 | 换的代价 |
|------|---------|---------|---------|
| PostgreSQL | 已有业务数据和审计数据 | 能换，但没理由换 | 数据迁移 + 运维成本 |
| ChromaDB | 用于 JD 向量检索 | 能换成 pgvector（推荐）/ LanceDB | pgvector 并发性能更优（100 并发下快 2.4 倍）且无需额外进程，但 ChromaDB 开发体验更好 |
| OpenClaw Gateway | V1 已接入飞书/微信，7 个 Skill 已发 ClawHub | 能去掉，或改为兼容适配层 | 需补齐渠道接入与技能分发策略 |
| Pydantic Structured Output | 类型安全的 LLM 输出 | 能换成 JSON mode + 手工解析 | 代码量增加 |
| LangGraph | 当前主流程不依赖 | 可按需引入（仅审批/长事务场景） | 增加复杂度与依赖管理 |

**对新功能的约束宽松度**：

| 新功能 | 是否必须 Patchright | 是否必须浏览器 | 最简方案 |
|--------|-------------------|--------------|---------|
| 面经采集（牛客） | 看是否有反爬 | 可能需要 | Patchright 或 requests |
| 面经采集（脉脉） | 看是否有反爬 | 可能需要 | Patchright 或 requests |
| 面经采集（小红书） | 反爬严格 | 大概率需要 | Patchright |
| 技术雷达（RSS） | 不需要 | 不需要 | requests + feedparser |
| 技术雷达（GitHub Trending） | 不需要 | 不需要 | requests + BeautifulSoup |
| 技术雷达（公众号） | 反爬严格 | 需要 | Patchright |
| 内容发布（公众号 API） | 不需要 | 不需要 | WeChat MP API（HTTP） |

**结论：Patchright 和浏览器不是所有功能的约束。RSS/API 类数据源不需要浏览器。架构应支持"有浏览器"和"无浏览器"两类采集器共存。**

### 0.7 需求与架构的映射关系

从需求中提炼出架构必须解决的核心问题：

| 核心问题 | 来源需求 | 架构含义 |
|---------|---------|---------|
| **领域无关的框架** | 通用可扩展性 P0 + 领域无关性 P0 | 核心框架不能出现"boss""job""resume"等求职术语。求职是模块，不是框架 |
| **模块自包含** | 功能独立性 P0 | 一个模块 = 一个目录，包含自己的采集、处理、路由、存储、配置、调度声明 |
| **多源异构采集** | F1(牛客/脉脉/小红书) F2(RSS/GitHub/公众号) | 可插拔的采集器。有的要 Patchright，有的只要 HTTP，架构不假设所有采集都需要浏览器 |
| **浏览器按需池化** | Phase1(BOSS) + F1(反爬站点) + F2(公众号) | 需要浏览器的模块从 Pool 获取；不需要的模块完全不碰浏览器 |
| **统一 LLM 调用** | 所有功能都用 LLM | 统一的 LLM 路由（按任务选模型、主备降级、token 预算） |
| **调度灵活性** | 不同模块不同调度策略 | 调度器接受模块自注册的任务 + 多种模式（周期/定时/手动） |
| **信息治理** | F1 F2 去重/过滤/压缩 | 通用组件，可复用于任何信息采集场景 |
| **砍死代码** | 4 个不用的功能 | V2 移除，减少维护负担 |
| **一次定义不重建** | 未来深度调研/旅游/财务等新场景 | 框架层只定义 Module 接口 + Capability 层，业务逻辑全在模块内。新场景 = 新模块，框架不动 |

### 1.2 关键架构模式提炼

从上述 10+ 个项目中，提炼出 4 种核心架构模式：

#### 模式 A：Gateway-Centric（OpenClaw 原生路线）

```
用户消息 → Gateway(TypeScript) → Brain → Skill → exec/Lobster/内置Tool
```

**代表**：JobClaw、internship-scout、gstack workflows

**核心思路**：一切围绕消息网关，功能通过 Skill(Markdown) + Plugin(TypeScript) 扩展。

| 优势 | 劣势 |
|------|------|
| 生态最大（329K star）、16+ 渠道原生 | TypeScript 栈，Python 库无法使用 |
| Heartbeat/Cron 调度成熟 | 内置 Playwright 无 binary-level 反检测 |
| ClawHub 社区分享 | Lobster 步骤是 CLI 子进程，无法共享浏览器 |

**对我们的适用性：❌ 淘汰。** Patchright 是 Python 库；CLI 子进程无法共享浏览器会话。

#### 模式 B：Extensions-Only Nano-Kernel（Yodoca 路线）

```
Nano-Kernel（发现 + 加载 + 路由 + 事件循环）
  ↓
Extensions（按 Protocol 分类）
  ├── ChannelProvider   → 消息收发
  ├── ToolProvider      → 工具能力
  ├── AgentProvider     → 智能体
  ├── SchedulerProvider → 定时任务
  └── ContextProvider   → 上下文增强
```

**核心思路**：内核极小（只做发现、加载、事件分发），所有功能都是 Extension，通过 Protocol 约束。声明式 Agent（manifest.yaml），不写 Python 也能创建 Agent。

| 优势 | 劣势 |
|------|------|
| 最干净的关注点分离 | 极早期（18 star），缺乏实战验证 |
| 内核永远不需要改 | 过度抽象增加开发摩擦（要理解 7 种 Protocol） |
| 社区可贡献 Extension | Extension 间通信通过 ExtensionContext 间接调用，调试困难 |
| SQLite 零外部依赖 | 无浏览器池、无 LLM 路由，这些都要自建 |

**对我们的适用性：⚠️ 架构理念好，但工程不成熟。** Protocol-based 类型发现很优雅，但 7 种 Protocol 对我们过度——我们的 Module 本质就是"采集→处理→推送"的 Pipeline，不需要区分 ChannelProvider / ToolProvider / AgentProvider。

#### 模式 C：Extension + Chain + Provider（AGiXT 路线）

```
Core（API Server + Agent Loop + Memory）
  ↓
Extensions（bolt-on 能力模块）
  ↓
Chains（多步骤工作流编排）
  ↓
Providers（LLM / TTS / Vision / ...）
```

**核心思路**：三层扩展架构。Extension 提供能力，Chain 编排流程，Provider 提供 AI。社区通过 Hub 分享 Extension/Chain/Prompt。

| 优势 | 劣势 |
|------|------|
| 成熟、功能丰富（3.2K star） | Docker 多容器部署（Core + DB + Redis + ...） |
| Hub 社区生态 | 学习曲线陡峭（Extension 要理解 SDK/JWT/递归控制） |
| 多 AI Provider 开箱即用 | 浏览器进程内共享不可能（Docker 隔离） |

**对我们的适用性：❌ 淘汰。** Docker 多容器对个人项目过重；浏览器进程内共享无法实现。

#### 模式 D：Modular Monolith + Capability Layer

```
API Server（自动发现 Module → 注册路由/任务/Schema）
  ↓
Modules（每个功能一个目录，自包含 Pipeline）
  ↓
Capabilities（Browser Pool / LLM Router / Storage / Notifier / Scheduler）
```

**核心思路**：单进程，Module 通过接口隔离。Module 自声明路由、定时任务、DB Schema。Capability 层提供共享基础设施。不需要插件加载器，Python import 即可。

| 优势 | 劣势 |
|------|------|
| 最简单、最直接、调试最容易 | Module 隔离靠纪律约束而非框架强制 |
| 单进程：浏览器、LLM、调度全共享 | 没有 Plugin Runtime 的"酷感" |
| pip install 友好 | 需要自建注册机制 |
| 社区贡献门槛最低（写个 Python 目录） | — |

**对我们的适用性：✅ 最匹配。** 满足所有需求，工程最简。

### 1.3 四种模式的需求覆盖度检验

| 需求（来自 0.7） | A: Gateway | B: Nano-Kernel | C: Extension+Chain | D: Modular Monolith |
|-----------------|-----------|---------------|-------------------|-------------------|
| 领域无关框架 | △ 部分求职代码在核心 | ✓ 内核零业务 | ✓ 核心通用 | ✓ core/ 无业务词汇 |
| 模块自包含 | △ Skill=Markdown | ✓ Extension 目录 | ✓ Extension | ✓ Module 目录 |
| Python 运行时 | ✗ TypeScript | ✓ | ✓ | ✓ |
| Patchright 反检测 | ✗ 只有 Playwright | 需自建 Extension | ✗ Docker 隔离 | ✓ Capability 原生 |
| 浏览器进程内共享 | ✗ CLI 子进程 | 取决于实现 | ✗ Docker 隔离 | ✓ 单进程 |
| 浏览器按需 | — | ✓ | — | ✓ Module 自选 |
| 多源异构采集 | △ | ✓ | ✓ | ✓ |
| 调度灵活 | ✓ Heartbeat/Cron | ✓ SchedulerProvider | △ Chain 触发 | ✓ Module 自注册 |
| 新场景=新目录 | △ 需改后端 | ✓ | ✓ | ✓ |
| 单机部署 | ✓ | ✓ | ✗ Docker 多容器 | ✓ |
| pip 可安装 | ✗ npm | 可以 | ✗ | **可以** |
| 一次定义不重建 | ✗ | ✓ | ✓ | ✓ |

### 1.4 最终选择：模式 D，吸收 B 的精华

**选 D（Modular Monolith + Capability Layer）**，同时从 B（Nano-Kernel）吸收以下理念：

| 从 B 吸收的理念 | 如何落地 |
|---------------|---------|
| 内核零业务代码 | `core/` 不出现任何领域词汇 |
| Module 自声明能力 | BaseModule 接口严格定义 name / router / tasks / schemas |
| 自动发现 | `discover_modules()` 扫描 modules/ 目录 |
| 生命周期管理 | `on_startup()` / `on_shutdown()` 回调 |

**不采用 B 的过度抽象**：

| B 的设计 | 我们不用的原因 |
|---------|-------------|
| 7 种 Protocol 类型 | 我们的 Module 就是 Pipeline，不需要区分 Channel/Tool/Agent/... |
| ExtensionContext 间接通信 | 直接 import Capability 更简单、更可调试 |
| 声明式 Agent (manifest.yaml) | 我们的 Module 逻辑需要 Python 代码，YAML 不够 |
| SQLite Event Journal | PostgreSQL 已是当前选择，没理由降级 |

### 1.5 爆款开源项目共性（用于反推架构）

结合 OpenClaw、n8n、Dify、ComfyUI 的官方文档与社区实践，爆款项目通常不是只靠“架构优雅”，而是靠“可扩展 + 可分发 + 可复用 + 可参与”四个飞轮同时成立。

| 共性机制 | 社区案例 | 对 OfferPilot 的架构要求 |
|---------|---------|------------------------|
| **能力可声明** | OpenClaw 的 `SKILL.md`（YAML frontmatter + 指令）让模型“看见能力菜单” | Module 必须有 capability/trigger_examples，供 Router 读取 |
| **扩展可安装** | Dify 插件支持 Marketplace/GitHub/本地包安装，并带权限策略 | Module 包需要安装清单、版本、来源、权限声明 |
| **模板可复制** | n8n 有模板库与自定义模板 API（可自建私有模板源） | OfferPilot 需要 template packs（场景模板）和导入机制 |
| **生态可贡献** | ComfyUI Registry 把 custom nodes 变成可检索资产 | Module Registry + 官方索引页 + 贡献规范 |
| **路由可预测** | OpenClaw channel routing 强调“主机策略决定路由，模型不选渠道” | 渠道路由必须 deterministic，防误发、可审计 |
| **上手零摩擦** | 一键启动、开箱即用、最短路径见到结果 | 3 分钟 Quick Start + 示例场景一键跑通 |

> 结论：真正能长期增长的项目，不是“一个好框架”，而是“一个能让别人快速做出结果并愿意贡献的生态系统”。

## 二、OpenClaw 深度分析：为什么火、我们该学什么

> 不是问"V1 用了 OpenClaw 的什么"，而是问"OpenClaw 为什么 329K star（2026.3 GitHub 总榜第一），它的哪些设计理念对我们有价值"。

### 2.1 OpenClaw 为什么火（329K star 的核心原因）

| 原因 | 具体表现 | 我们能学到什么 |
|------|---------|-------------|
| **"做事的 AI"** | 不只是聊天，后台持久运行，自动执行多步任务："ChatGPT 给你菜谱，OpenClaw 帮你下单买菜" | 我们的 Pipeline 定时执行 + 自动推送本质上就是这个模式 |
| **Skills 生态** | 5,700+ 社区技能，SKILL.md 自然语言定义能力。LLM 每个推理循环读"菜单"（SKILLS.md），自主决定调用哪个 Skill | **核心启发**：Module 应该有能力声明，让 LLM 能"看到"并路由 |
| **多渠道统一** | 16+ 渠道（WhatsApp/Telegram/Discord/Slack/飞书/微信），ChannelPlugin 三层抽象 | Channel 抽象接口必须保留 |
| **零摩擦自托管** | 一键安装，本地运行，Mac/Linux/Windows/树莓派 | `pip install offerpilot && offerpilot start` 对标 |
| **MIT 开源 + 社区** | 1,075+ 贡献者，完全透明 | 架构必须对社区友好，新人能快速贡献 Module |
| **Claude Max 集成** | 用 setup token 接入，无需按 token 付费 | 参考：LLM Provider 灵活配置 |

### 2.2 OpenClaw 的 3 个核心设计模式（必须理解）

#### 模式 1：Skills — LLM 可读的能力声明

OpenClaw 的 Brain 推理循环：

```
用户发消息 → Brain 加载上下文：
  ├── SOUL.md（Agent 人格）
  ├── SKILLS.md（所有已安装能力的"菜单"）
  ├── TOOLS.md（底层工具列表）
  └── 对话历史 + 记忆
→ LLM 看到菜单，决定调用哪个 Skill
→ Skill 执行
→ 结果追加到上下文
→ 继续推理直到完成
```

**SKILL.md 的本质**：一个给 LLM 看的"能力说明书"。YAML 元数据 + Markdown 指令。LLM 读这个说明书来判断"面对用户的请求，我该用哪个 Skill"。

**为什么这很重要**：用户说"帮我查一下字节跳动最近的面经"，系统需要知道 intel_interview 模块能处理这个请求。如果 Module 没有能力声明，消息路由就退化为硬编码的关键词匹配。

#### 模式 2：Channel — 多渠道消息抽象

```
Platform Protocol 层（飞书 WebSocket / Telegram Bot API / Discord Gateway / ...）
  ↓
Channel Abstraction 层（统一的消息格式 + 收发接口）
  ↓
Agent/Router 层（意图识别 → 路由到功能模块）
```

**ChannelPlugin 核心接口**：
- 配置 + 账户解析
- 消息接收（inbound）
- 消息发送（outbound）
- 安全策略 + DM 处理
- 群组/线程管理
- 媒体支持

**为什么这很重要**：今天只用飞书，但架构要支持未来一行配置加 Telegram。如果消息处理硬编码了飞书 API，加渠道就要改框架。

#### 模式 3：持久后台运行 + 定时自动化

OpenClaw 不是"用完即走"的工具，而是一个**常驻后台的 Agent 服务**：

- Heartbeat/Cron 定时触发任务
- Memory 跨会话持久记忆
- AGENTS.md 定义自主行为规则

**这个我们已经完全覆盖**：Scheduler（从 ProductionGuard 演进）+ PostgreSQL 持久化 + 模块定时任务。

### 2.3 缺口识别（与 OpenClaw 核心模式对照）

| OpenClaw 特性 | v2.0 状态 | 现状说明 | 风险 |
|-------------|-----------|---------|------|
| **Skills（LLM 可读能力声明）** | ⚠️ **缺失** | Module 有 name/description，但没有给 LLM 看的能力描述和触发示例 | 消息路由退化为关键词硬编码 |
| **Brain 推理循环（意图路由）** | ⚠️ **缺失** | 没有设计消息 → Module 的路由机制 | 用户发消息不知道该调哪个 Module |
| **多渠道 ChannelPlugin** | ⚠️ **部分** | `core/channel/` 预留了，但只说"未来加"，没有定义接口 | 加渠道时发现架构不支持 |
| **SOUL.md（Agent 人格）** | △ → Phase 3 | Phase 2 用系统 prompt 配置化；Phase 3 升级为 Core Memory SOUL Block（详见 §10.3.5） | Phase 3 实现 |
| **持久后台运行** | ✅ 完全覆盖 | Scheduler + 定时任务 | — |
| **零摩擦安装** | ✅ 完全覆盖 | `pip install offerpilot && offerpilot start` | — |
| **社区可贡献** | ✅ 完全覆盖 | Module 即目录，新人写目录就行 | — |
| **ClawHub 生态** | ❌ 不保留 | 用 GitHub + PyPI 替代 | 失去 OpenClaw 社区流量，但获得独立品牌 |

**关键发现：我们缺了两个重要的东西**：

1. **Module 能力声明**（相当于 SKILL.md）— Module 不只是代码，还需要一份"给 LLM 看的说明书"
2. **消息意图路由**（相当于 Brain）— 用户发来消息时，系统如何决定交给哪个 Module 处理

### 2.4 补齐缺失：Module 能力声明 + 消息路由

#### Module 能力声明

每个 Module 除了 name/description/router/tasks 之外，还应声明：

| 声明项 | 用途 | 示例（intel_interview） |
|--------|------|----------------------|
| `capability` | 给 LLM 看的能力描述（一段话） | "我可以从牛客、脉脉、小红书采集面经情报，提取面试轮次、考察点、编程题，生成每日面经简报" |
| `trigger_examples` | 用户可能说的触发语句 | ["查看今天的面经", "字节跳动最近面试都考什么", "面经日报"] |
| `response_hint` | 回复风格提示 | "返回面经简报卡片，包含公司、轮次、高频考点" |

这就是 SKILL.md 的 Python 等价物。不是 Markdown 文件，而是 Module 接口的一部分。

#### 消息意图路由

```
用户消息（飞书/CLI/API）
  ↓
消息路由器（core/router.py）
  ├── 收集所有 Module 的 capability + trigger_examples
  ├── 构建系统 prompt："以下是可用的功能模块：..."
  ├── LLM 判断意图 → 返回 module_name + 参数
  └── 调用对应 Module 的消息处理方法
  ↓
Module 处理 → 返回结果
  ↓
通过原渠道回复用户
```

**对比 OpenClaw Brain**：
- OpenClaw Brain 加载 SKILLS.md + TOOLS.md → LLM 推理循环 → 调用 Skill
- 我们的路由器加载 Module capabilities → LLM 意图识别 → 调用 Module

**本质相同，实现更轻量**：我们不需要 Brain 的 9 层策略引擎，只需要一次 LLM 调用做意图路由。

#### Channel 抽象接口

`core/channel/` 必须定义清晰的 Channel 接口：

| 接口方法 | 职责 |
|---------|------|
| `start()` | 启动渠道连接（飞书 WebSocket / Telegram 轮询 / ...） |
| `on_message(callback)` | 注册消息回调 |
| `send_text(target, text)` | 发送文本消息 |
| `send_card(target, card)` | 发送富文本/卡片消息 |
| `stop()` | 关闭连接 |

飞书作为默认实现。未来加 Telegram 只需新建 `core/channel/telegram.py` 实现同一接口。

### 2.5 修正后的 OpenClaw 决策

之前的结论"去掉 OpenClaw"方向没问题，但分析不够深入。修正为：

**我们不是"去掉 OpenClaw"，而是"用 Python 原生实现 OpenClaw 的核心设计模式"：**

| OpenClaw 设计模式 | 我们的 Python 实现 | 状态 |
|-----------------|------------------|------|
| **Skills（能力声明）** | Module 声明 capability / trigger_examples / response_hint | 新增到 Module 接口 |
| **Brain（意图路由）** | core/router.py — LLM 读 Module 能力列表 → 意图分类 → 路由 | 新增核心组件 |
| **ChannelPlugin（多渠道）** | core/channel/ — BaseChannel 接口 + 飞书默认实现 | 明确接口定义 |
| **SOUL.md（人格）** | 系统 prompt 配置模板 | 配置文件化 |
| **Heartbeat/Cron（调度）** | Scheduler（时段感知 + Module 自注册） | 已有 |
| **Memory（记忆）** | PostgreSQL + ChromaDB | 已有 |
| **ClawHub（生态）** | GitHub + PyPI + docs/module-guide.md | 已有 |

**结论：不依赖 OpenClaw 运行时，但完整保留了它最有价值的设计理念，用 Python 原生方式实现。**

### 3.1 设计原则

| 原则 | 含义 | 来源需求 |
|------|------|---------|
| **内核零业务** | `core/` 中不出现任何领域词汇（boss/job/resume/面经...） | 领域无关性 P0 |

| **配置驱动** | 所有阈值/频率/源/开关通过 .env 和配置文件控制 | 配置驱动 P1 |
| **一次定义不重建** | Module 接口 + Capability 接口稳定后，新增功能只加目录 | 通用可扩展 P0 |
| **不提前引入不需要的** | 当前不用的框架/依赖/抽象一律不加 | 务实 |

### 3.3 三层架构的职责边界

| 层 | 职责 | 不做什么 |
|----|------|---------|
| **接入层** | 接收外部请求（HTTP/飞书消息/CLI），路由到对应 Module | 不含任何业务逻辑 |
| **Module 层** | 实现具体业务功能（采集→处理→执行→通知的 Pipeline） | 不直接操作底层资源（浏览器/数据库连接），通过 Capability 使用 |
| **Capability 层** | 提供共享基础设施（浏览器池、LLM、存储、通知、调度） | 不含任何业务逻辑，不知道 Module 的存在 |

### 3.4 Module 概念

每个 Module 是一个**自包含的功能单元**，以目录形式存在于 `modules/` 下。

**Module 声明项**：

| 声明项 | 含义 | 是否必须 |
|--------|------|---------|
| `name` | 唯一标识，同时作为 API 路由前缀和日志 tag | 必须 |
| `description` | 模块描述，展示在 CLI 和 API | 可选 |
| `capability` | **给 LLM 看的能力描述**（一段话，说明这个模块能做什么） | 可选（有消息交互的 Module 必须提供） |
| `trigger_examples` | **触发示例**（用户可能说的话，帮助 LLM 做意图路由） | 可选（配合 capability 使用） |
| `router` | FastAPI 路由（HTTP API 接口） | 可选 |
| `scheduled_tasks` | 定时任务列表（名称、handler、调度模式、开关） | 可选 |
| `message_handler` | **消息处理方法**（当用户消息路由到此 Module 时调用） | 可选 |
| `db_schemas` | 建表 SQL，启动时自动执行 | 可选 |
| `on_startup` / `on_shutdown` | 生命周期回调 | 可选 |

**能力声明示例（相当于 OpenClaw SKILL.md 的 Python 等价物）**：

| Module | capability（给 LLM 看的） | trigger_examples |
|--------|------------------------|-----------------|
| boss_greet | "搜索 BOSS 直聘岗位，根据匹配度自动打招呼投递简历" | ["帮我搜后端岗位", "投递字节跳动", "开始投简历"] |
| boss_chat | "监控 BOSS 直聘聊天消息，智能识别意图并自动回复 HR" | ["查看聊天状态", "最近有 HR 联系吗"] |
| intel_interview | "从牛客、脉脉、小红书采集面经，提取考点，生成面经简报" | ["今天的面经", "字节最近面试考什么", "面经日报"] |
| intel_techradar | "监控技术动态，生成每日技术简报" | ["今天有什么技术新闻", "最近 AI 有什么新进展"] |
| email_tracker | "监控邮箱，识别面试邀请/结果/日程" | ["邮件有新的面试通知吗", "查看邮件状态"] |

Module 的能力声明不是 Markdown 文件，而是 Module 接口的一部分。消息路由器读取所有 Module 的 capability 构建"能力菜单"，让 LLM 做意图分类。

**Module 的 Pipeline 统一范式**：

```
触发（Scheduler 定时 / API 请求 / 飞书消息）
  → 采集（Patchright 爬取 / HTTP API / RSS）
  → 处理（LLM 分析 / 去重 / 过滤 / 聚合）
  → 执行（浏览器操作 / API 调用 / 写入存储）
  → 通知（飞书推送 / SSE 事件 / 日志）
```

不同 Module 的差异只在每个阶段的具体逻辑，范式不变。

**Module 之间的关系**：低耦合、数据约定。Module A 不 import Module B 的代码。跨模块数据共享通过共享存储（PostgreSQL/ChromaDB）间接访问，以**约定的表结构和向量 collection 名称**为契约。例如 intel_query 检索的向量由 intel_interview 和 intel_techradar 写入，三者约定 collection 名和 embedding 维度，但不共享代码。

**Module 实现阶段**：

| Module | 阶段 | 说明 |
|--------|------|------|
| boss_greet | **Phase 2**（从 V1 迁移） | BOSS 投递核心功能 |
| boss_chat | **Phase 2**（从 V1 迁移） | BOSS 聊天核心功能 |
| email_tracker | **Phase 2**（从 V1 迁移） | 邮件追踪 |
| intel_interview | **Phase 2**（新建） | 面经情报流 F1 |
| intel_techradar | **Phase 2**（新建） | 技术雷达 F2 |
| intel_query | **Phase 2**（新建） | 语义检索 F3（依赖 F1/F2 数据） |
| feedback_loop | **Phase 3（基础已落地）** | 反馈闭环 F5 已接入基础偏好学习；更深演化增强继续迭代 |

### 3.5 Capability Layer 职责划分

> **原则：不提前引入不需要的。** 下表标注了每个 Capability 的实现阶段。Phase 2 标"必须"的立即实现；Phase 3 标"预留"的只定义接口、不写实现。

| Capability | 职责 | 关键设计点 | 阶段 |
|-----------|------|-----------|------|
| **Browser Pool** | 管理 Patchright 浏览器实例，按站点复用，反检测配置 | 按 site_key 复用 persistent_context；不需要浏览器的 Module 完全不碰此 Capability；Cookie 过期检测；健康检查与自动恢复 | **Phase 2 必须** |
| **LLM Router** | 统一 LLM 调用，按 task_type 选模型，主备降级 | 任务类型 → (主模型, 备模型) 映射；Pydantic Structured Output 封装；token 预算；重试 + 降级 | **Phase 2 必须** |
| **Storage Engine** | PostgreSQL 连接池 + ChromaDB 向量存储 | 异步连接池；Module 声明表结构，启动时自动迁移；向量嵌入 + 检索 | **Phase 2 必须** |
| **Notifier** | 主动推送通知（无需用户发起对话） | 抽象接口 send(level, content)；飞书卡片模板（info/warning/critical 分级）；通用 Webhook。**与 Channel 的区别**：Channel 处理"用户发消息→系统回复"双向交互；Notifier 处理"系统主动推送→用户"单向通知（如日报、告警、调度结果）。两者底层可共享飞书 API，但语义不同 | **Phase 2 必须** |
| **Scheduler** | 时段感知定时调度 | 从 ProductionGuard 演进；Module 自注册任务；interval/daily/manual 模式；高峰/低峰自适应；资源治理 | **Phase 2 必须** |
| **Channel** | 多渠道消息收发，统一接口抽象 | BaseChannel 接口（start/on_message/send_text/send_card/stop）；飞书默认实现（lark_oapi WebSocket）；未来加渠道只需新增实现。**关键约束**：飞书 WebSocket SDK 要求 3 秒内 ACK 消息，因此 on_message 必须立即返回，耗时操作（如 Router LLM 调用）须异步投递到后台任务 | **Phase 2 必须** |
| **Router** | 消息意图路由，用户消息 → Module | 收集 Module 的 capability 构建"菜单"→ LLM 一次调用分类 → 路由到 Module 的 message_handler | **Phase 2 必须** |
| **Event Bus** | 进程内事件分发 + SSE 外推 | Pipeline 阶段事件 emit；前端 SSE 实时展示 | **Phase 2 必须** |
| **Config** | 统一配置管理 | Pydantic Settings；Module 声明 config schema；类型安全 + 校验 | **Phase 2 必须** |
| **Policy Engine** | 动作风险分级与审批门控 | 统一策略：`safe/confirm/blocked`；高风险动作（投递/外发）支持二次确认；规则可配置且可审计 | **Phase 2 简版**（仅 safe/confirm/blocked 三级 + 审计日志） |
| **Observability** | 指标、追踪、审计日志 | 统一 trace_id；关键指标（成功率/时延/成本）；错误分级告警；SLO 面板 | **Phase 3 预留**（Phase 2 仅结构化日志 + trace_id） |
| **Template Registry** | 模板分发与复用 | 提供官方模板索引、版本化、依赖声明；支持导入导出场景模板 | **Phase 3 预留**（Phase 2 不实现） |

### 3.6 消息接入与意图路由（替代 OpenClaw Brain + Skills 的核心设计）

**两种触发模式并存**：

| 触发模式 | 场景 | 机制 |
|---------|------|------|
| **定时触发** | 定时采集面经、巡检聊天、拉取邮件 | Scheduler 调用 Module 的 scheduled_task handler |
| **消息触发** | 用户说"查看面经"、"搜索岗位" | Channel 接收消息 → Router 路由到 Module 的 message_handler |

**消息路由流程（对标 OpenClaw Brain 推理循环）**：

```
用户消息（飞书/CLI/API）
  ↓
Channel 层接收，统一格式化
  ↓
消息路由器（core/router.py）
  ├── 1. 收集所有 Module 的 capability + trigger_examples
  ├── 2. 构建 prompt："你是 OfferPilot 消息路由器。以下是可用功能：
  │       - boss_greet: 搜索 BOSS 直聘岗位...
  │       - intel_interview: 从牛客、脉脉采集面经...
  │       - ...
  │       用户说了：{message}
  │       请判断应该路由到哪个 Module（或无匹配）"
  ├── 3. LLM 返回 module_name + 提取的参数
  └── 4. 调用对应 Module 的 message_handler(params)
  ↓
Module 处理 → 返回结果
  ↓
通过原渠道回复用户
```

**对比 OpenClaw Brain**：
- OpenClaw：Brain 加载 SKILLS.md → 多轮推理循环 → 9 层策略引擎 → 调用 Skill
- 我们：路由器加载 Module capabilities → **一次 LLM 调用**做意图分类 → 调用 Module

本质相同，实现更轻量。Phase 2 不需要 Brain 的多轮推理，因为 Module 的 message_handler 内部已经有完整的业务逻辑。Phase 3 将 Router 升级为 Brain（ReAct 推理循环），支持多步推理和 Tool Call，详见第九节。

**路由降本策略**：并非每条消息都需要 LLM 推理。实际路由分三级：
1. **精确匹配**（零成本）：用户点击飞书卡片按钮 / API 指定 module_name → 直接路由
2. **前缀匹配**（零成本）：trigger_examples 编译为关键词 trie，命中则直接路由
3. **LLM 推理**（兜底）：前两级未命中时调用 LLM 做意图分类

大部分日常交互（点卡片按钮、固定命令）走 1-2 级，LLM 仅处理模糊自然语言，成本可控。

**路由失败处理**：当三级路由均无法匹配时，Router 返回友好提示（“我目前支持以下功能：...”列出可用 Module）而不是静默失败。Phase 3 升级为 Brain 后，未匹配的请求可尝试通过 Tool Call 或 MCP 处理，“无法处理”的情况将大幅减少。

**渠道路由原则（借鉴 OpenClaw）**：
- 模型只决定“调用哪个 Module”，**不决定“从哪个渠道发送”**
- 回复渠道由 host 的 deterministic routing 决定：默认回原渠道，可配置强制规则（例如某些告警只发飞书）
- 所有出站消息记录 `source_channel`、`target_channel`、`route_reason`，确保可审计

**Channel 抽象接口（对标 OpenClaw ChannelPlugin）**：

| 接口方法 | 职责 |
|---------|------|
| `start()` | 启动渠道连接（飞书 WebSocket / Telegram 轮询 / ...） |
| `on_message(callback)` | 注册消息回调（收到消息时调用路由器） |
| `send_text(target, text)` | 发送文本消息 |
| `send_card(target, card)` | 发送富文本/卡片消息 |
| `stop()` | 关闭连接 |

飞书作为默认实现（`lark_oapi` WebSocket 长连接，不需要公网 IP）。未来加 Telegram 只需新建一个 Channel 实现文件。

**飞书 WebSocket 关键约束**：`lark_oapi` SDK 要求收到消息后 **3 秒内 ACK**，否则触发超时重试。因此 Channel 的 `on_message` 回调必须立即 ACK 并将消息投递到 `asyncio.Queue`，由独立协程异步执行 Router → Module → 回复的完整链路。这意味着消息处理是 **"即时 ACK + 异步执行 + 异步回复"** 模式，而非同步阻塞。

### 3.7 编排策略

**核心判断**：LangGraph 的真实价值是 Checkpoint 持久化 + interrupt_before 人机协作。V1 的 4 个 workflow 中只有材料审批真正用到了这个能力，其余都是纯 Python 函数套了个图的壳子。

**原则：用最简单的方案。不为了用框架而用框架。**

| 工作流 | 选型 | 理由 |
|--------|------|------|
| BOSS 打招呼 | 纯 Python async | 线性流：搜索→分析→打招呼，for/if 足够 |
| BOSS 聊天 | 纯 Python async | 四层门控是 if/else 逻辑 |
| 面经情报 | 纯 Python async + asyncio.gather | 多源并发采集→去重→分析→聚合→推送 |
| 技术雷达 | 纯 Python async + asyncio.gather | 同上 |
| 语义检索问答（F3） | 纯 Python async | Query → 向量检索 → 重排 → 引用回答，链路短且可控 |
| 邮件分类 | 纯 Python async | 拉取→分类→更新，3 步线性 |
| 反馈闭环（F5） | 纯 Python async（批处理） | Phase 3：收集反馈→更新权重→离线回放评估，不需复杂状态机 |
| 高风险动作（投递/外发） | 纯 Python + Policy Engine | 执行前统一做策略判定（safe/confirm/blocked） |
| 内容发布(Phase 3) | 到时再评估（可引入状态机） | Phase 3：若出现长事务审批/恢复，再按需引入 LangGraph |

所有 Pipeline 遵循统一范式：`async def run()` 作为入口，Scheduler 和 API 都调用同一个函数。每个 Stage 完成后 emit 事件通知进度。单条失败不阻断批量。

## 六、GitHub 项目品质建设

### 6.1 品牌定位

> **OfferPilot** — 通用个人智能体平台（Personal Agent Platform）。
> Pipeline 重型自动化（定时采集、LLM 处理、自动执行）+ Brain 轻量级交互（闹钟、天气、日程）+ MCP 生态接入。
> 当前内置求职自动化模块（BOSS 直聘反检测 + 面经情报 + 技术雷达），支持任意场景扩展。
> 纯 Python 单进程，pip install 即用。Windows / Linux / macOS 原生运行，Docker 一键部署。

### 6.2 差异化卖点

| 卖点 | 说明 | 竞品状态 |
|------|------|---------|
| **Patchright 反检测浏览器池** | 唯一内置 binary-level 反检测的 AI 自动化平台 | AGiXT/OpenBotX/Yodoca 均无 |
| **纯 Python 单进程** | 不依赖 Node.js/Docker/任何外部运行时；Windows/Linux/macOS 原生运行 + Docker 可选 | AGiXT 要 Docker，OpenClaw 是 TypeScript |
| **模块即目录** | 新建目录 = 新功能，零框架配置 | AGiXT 需 SDK，Yodoca 需 Protocol |
| **中文生态原生** | 开箱支持 BOSS 直聘、牛客、脉脉、飞书 | 全球项目无中文平台支持 |
| **Hybrid: Pipeline + Brain** | 重型任务用 Pipeline（采集→处理→执行→通知），轻量任务用 Brain ReAct 循环 + Tool Call | OpenBotX 只有对话，AutoGPT 只有 Agent |
| **MCP 双向集成** | 对内接入 10,000+ 外部工具；对外暴露 Tool 让 Cursor/Claude 调用 | 竞品无 MCP 支持 |
| **完全独立** | 不依赖 OpenClaw/LangChain 等重框架 | 多数项目绑定特定框架 |

## 九、Phase 3-4 远景：通用个人智能体架构（深度调研与设计）

> 本节是对 Phase 3/4 架构的**深度调研和完整设计**，目的是确保 Phase 2 的架构底座能够**无缝升级**为 Jarvis 级通用智能体，而不需要重写。

### 9.1 核心问题：为什么 Pipeline-Only 架构不够？

Phase 2 的架构是：

```
Channel → Router（一次 LLM 分类）→ Module（Pipeline: 采集→处理→执行→通知）
```

这个架构有一个根本性的局限：**所有能力都必须是 Module（Pipeline）**。

| 用户说 | 需要什么？ | Pipeline 能做吗？ |
|--------|---------|----------------|
| "帮我搜后端岗位" | Module: boss_greet Pipeline | ✅ 可以 |
| "今天的面经" | Module: intel_interview Pipeline | ✅ 可以 |
| "设个明天早上7点的闹钟" | 一个函数调用 `set_alarm(7:00, "起床")` | ❌ 杀鸡用牛刀 |
| "今天北京天气怎么样" | 一个 API 调用 `get_weather("北京")` | ❌ 不需要 Pipeline |
| "播放我的收藏歌单" | 一个 Spotify API 调用 | ❌ 不需要 Pipeline |
| "帮我查今天天气然后安排下午出行" | 两步推理：查天气 → 据此创建日程 | ❌ Router 只能路由到一个 Module |
| "我需要一个每天监控 BTC 的功能" | LLM 生成新代码 → 热加载 | ❌ 完全超出能力 |

**结论**：要成为 Jarvis，必须同时具备三种能力执行模式——轻量级 Tool Call、重型 Pipeline、多步推理编排。

### 9.2 业界深度调研：通用个人智能体是怎么做的？

#### 9.2.1 OpenClaw 的 Brain + Hands 架构（329K Star，2026 年 GitHub 总榜第一）

OpenClaw 将系统分为两个根本性的组件：

| 组件 | 职责 | 关键设计 |
|------|------|---------|
| **Brain（大脑）** | LLM 推理层，接收消息、加载记忆、多轮推理、决定行动 | **永远不直接接触文件系统和网络**——只通过 tool_call 请求 Hands 执行 |
| **Hands（双手）** | 执行层，提供 4 种基础能力：Shell、Filesystem、Browser、HTTP | 通过结构化 tool-call 协议接收请求，返回带 exit code 的结果 |

**Brain 推理循环的 6 步实现**：

```
① Load → 加载会话历史 + 记忆文件（SOUL.md）+ 系统提示词
② Call → 将上下文 + 可用工具列表发送给 LLM
③ Parse → LLM 返回文本（结束循环）或结构化 tool_call
④ Execute → 验证参数 + 调用对应 Skill/Tool
⑤ Append → 工具结果追加到上下文
⑥ Loop → 回到 ②，直到 LLM 产生最终文本回复
```

**关键洞察**：
- TOOLS.md 和 SKILLS.md 显式控制 Brain 的能力边界——"菜单式"安全模型
- Skill 系统支持热加载（`load.watch: true`，250ms debounce）
- 3,000+ 社区 Skills 通过 ClawHub marketplace 分发
- Meta-Skill `create-skill` 允许 Brain 在对话中 10 秒创建新 Skill

#### 9.2.2 Pincer 的 ReAct Loop + 安全优先架构（7,800 行 Python，安全标杆）

Pincer 是一个纯 Python 个人 AI 助手，以**安全和可审计**为第一优先级：

| 维度 | Pincer 设计 | OfferPilot 启示 |
|------|-----------|----------------|
| **推理模式** | ReAct Loop（同 OpenClaw Brain 6 步循环） | 采用相同模式，Brain 替代 Router |
| **工具系统** | 异构工具（日历/邮件/Shell/搜索/代码执行）统一注册 | Tool Registry 统一管理 |
| **记忆** | SQLite + FTS5（不用向量数据库，个人场景够用） | 已有 PostgreSQL + ChromaDB，更强 |
| **Skill 沙箱** | subprocess 隔离 + AST 扫描 + 网络域名白名单 + 文件系统限制 | 采用相同安全模型 |
| **成本控制** | 每日硬消费上限 + 实时追踪 + 自动降级模型 | Phase 3 引入 |

**关键洞察**：Pincer 证明了"7,800 行 Python 就能做一个完整的、安全的、多渠道个人 AI 助手"。不需要重框架。

#### 9.2.3 MCP（Model Context Protocol）—— AI 工具集成的 USB-C

MCP 是 Anthropic 捐赠给 Linux 基金会的开放标准，2026 年已被 OpenAI、Google、Microsoft 全面采纳：

| 特性 | 说明 |
|------|------|
| **定位** | AI 应用连接外部工具和数据源的统一协议，"AI 的 USB-C" |
| **工具发现** | 客户端连接 MCP Server 后自动获取可用工具列表和 JSON Schema |
| **传输协议** | stdio（本地进程）/ HTTP+SSE / Streamable HTTP |
| **生态规模** | 10,000+ 公开 MCP Server（awesome-mcp-servers 社区列表） |
| **Python 实现** | FastMCP：`@mcp.tool` 装饰器自动从函数 type hints 生成 JSON Schema |
| **双向价值** | OfferPilot 既是 MCP **Client**（接入外部工具）又是 MCP **Server**（将自身工具暴露给 Cursor/Claude 等） |

**MCP 对 OfferPilot 的战略意义**：
1. **不再需要为每个外部服务写代码**——接入 MCP Server 即可
2. **OfferPilot 的工具可以被其他 AI 产品使用**——一个 Tool 既服务 OfferPilot Brain 又服务 Cursor
3. **社区贡献门槛极低**——贡献一个 MCP Tool 只需写一个函数 + 装饰器

#### 9.2.4 Agents vs Workflows：2026 年共识

2026 年业界对"何时用 Agent 推理 vs 何时用 Pipeline 工作流"已形成明确共识：

| 模式 | 适用场景 | 特点 |
|------|---------|------|
| **Workflow（Pipeline）** | 所有执行路径在设计时可确定 | 可预测、低延迟、低成本、易调试 |
| **Agent（ReAct）** | 执行路径必须在运行时根据中间结果决定 | 适应性强、能处理模糊指令、延迟和成本更高 |
| **Hybrid（推荐）** | 顶层用确定性编排，特定步骤内用 LLM 推理 | 兼顾可靠性和灵活性 |

**OfferPilot 的混合策略**：
- **Pipeline（Module）**用于确定性的重型自动化（投递、采集、推送）—— 路径在设计时确定
- **Brain（ReAct）**用于用户实时对话中的不确定任务 —— 路径在运行时按需决定
- 两者共存，Brain 可以调用 Module 作为一个"高级工具"

#### 9.3.2 Brain vs Router：为什么必须升级

| 维度 | Router（Phase 2） | Brain（Phase 3） |
|------|------------------|-----------------|
| **决策模式** | 一次 LLM 调用，分类到一个 Module | ReAct 循环：Think → Act → Observe → Think → ... |
| **动作类型** | 只能路由到 Module | 可调用 Tool、Module、MCP Tool |
| **循环** | 无循环，fire-and-forget | 循环直到 LLM 产生最终文本 |
| **组合能力** | 不能 | "查天气→创日程→设提醒"一次对话完成 |
| **上下文** | 仅当前消息 | 会话历史 + Memory + 工具结果 |
| **能力发现** | Module capability 列表 | Tool + Module + MCP 统一能力列表 |
| **兼容性** | — | Router 是 Brain 的退化形式（循环只执行一轮） |

**Brain 的内部实现伪代码**：

```python
async def brain_loop(message: str, context: BrainContext) -> str:
    """ReAct reasoning loop - Phase 3 core"""
    tools = (
        tool_registry.list_tools()           # Ring 1: Built-in Tools
        + module_registry.as_tools()         # Ring 2: Modules exposed as tools
        + mcp_client.list_tools()            # Ring 3: External MCP Tools
    )
    
    context.append({"role": "user", "content": message})
    
    for step in range(MAX_TOOL_CALLS):
        response = await llm_router.call(
            messages=context.messages,
            tools=tools,
            model=select_model(context),
        )
        
        if response.is_text():               # Termination: LLM produces text
            return response.text
        
        for tool_call in response.tool_calls:
            result = await execute_tool(tool_call, context)
            context.append(tool_call_result(tool_call, result))
    
    return "Max tool calls reached."
```

> 注：上述伪代码展示 Brain 推理循环的核心逻辑。集成记忆系统后的完整版本（含记忆加载/写入阶段）见 §10.3.7。

**关键设计决策**：
1. **Brain 不是一个框架，是一个 < 200 行的 async 函数**。不引入 LangGraph/LangChain
2. **Module 暴露为 Tool**：`module_registry.as_tools()` 将每个 Module 的 `capability` 转换为标准 tool schema，Brain 调用时触发 Module 的 `message_handler`
3. **降本策略延续**：精确匹配 / 前缀匹配 / LLM 推理三级降本，Brain 只在 LLM 推理级别启动循环

#### 9.3.3 三环能力模型详解

**Ring 1: Tool（轻量级内置工具）**

| 特性 | 说明 |
|------|------|
| **定义** | 单个 Python async 函数 + type hints + docstring |
| **格式** | MCP 标准：使用 FastMCP `@mcp.tool` 装饰器，自动生成 JSON Schema |
| **执行** | 直接函数调用，毫秒级 |
| **状态** | 无状态，无生命周期 |
| **存储位置** | `tools/` 目录，每个文件一组相关 Tool |
| **注册方式** | 启动时扫描 `tools/` 目录，自动注册 |
| **双重身份** | 既是 Brain 的 Tool，又是对外的 MCP Server Tool |

Tool 示例：

```python
# tools/alarm.py
from offerpilot.core.tool import tool

@tool
async def set_alarm(time: str, label: str = "reminder") -> str:
    """Set an alarm. time format: HH:MM or YYYY-MM-DD HH:MM"""
    scheduler.add_one_shot(time, notify_user, args=[label])
    return f"Alarm set: {time} - {label}"

@tool
async def list_alarms() -> list[dict]:
    """List all pending alarms"""
    return scheduler.list_one_shots()
```

**Ring 2: Module（重型 Pipeline，Phase 2 原样保留）**

Module 不需要任何改动。Brain 通过 `module_registry.as_tools()` 将 Module 暴露为"高级工具"：

| Module 声明 | 转换为 Tool Schema |
|------------|------------------|
| `name: "boss_greet"` | `tool.name = "boss_greet"` |
| `capability: "搜索 BOSS 直聘岗位..."` | `tool.description = "搜索 BOSS 直聘岗位..."` |
| `trigger_examples: [...]` | 不暴露（Brain 推理能力足够） |
| `message_handler(params)` | `tool.function = message_handler` |

Brain 看到的统一工具列表中，`set_alarm` 和 `boss_greet` 形式完全相同——都是 `{name, description, parameters}` 的 tool schema。区别只在后端：一个调用函数，一个触发 Pipeline。

**Ring 3: MCP（外部工具生态）**

| 特性 | 说明 |
|------|------|
| **角色** | OfferPilot 作为 MCP Client 连接外部 MCP Server |
| **发现** | 启动时从 `mcp_servers.yaml` 读取配置，连接并获取工具列表 |
| **合并** | 外部工具与 Ring 1/Ring 2 合并为 Brain 的统一能力菜单 |
| **安全** | Policy Engine 门控：外部工具首次调用需用户确认 |

MCP 配置示例：

```yaml
# mcp_servers.yaml
servers:
  - name: google-calendar
    transport: stdio
    command: npx @anthropic/mcp-google-calendar
    auto_approve: [list_events, get_event]
    
  - name: filesystem
    transport: stdio
    command: npx @anthropic/mcp-filesystem /home/user/docs
    
  - name: custom-weather
    transport: http+sse
    url: http://localhost:8081/mcp
```

**Meta: Skill Generator（Phase 4）**

Skill Generator 是一个特殊的 Tool（Meta-Tool），它的作用是**创建新的 Tool 或 Module**：

```
用户："我需要一个每天监控 BTC 价格的功能"
  ↓
Brain 调用 Skill Generator（Meta-Tool）
  ↓
Skill Generator 内部流程：
  ① 分析需求 → "需要定时采集 + 推送" → 生成 Module（不是 Tool）
  ② LLM 生成代码：
     - modules/btc_monitor/__init__.py（Module 声明）
     - modules/btc_monitor/pipeline.py（采集→分析→推送）
  ③ AST 安全扫描：检查 import、检测 eval/exec、验证模式
  ④ subprocess 沙箱测试：隔离执行，5s 超时，256MB 内存
  ⑤ Policy Engine 确认："将创建新模块 btc_monitor，确认激活？"
  ⑥ 热加载：importlib.reload() + ModuleRegistry 重新扫描
  ↓
btc_monitor 立即可用，下次 Scheduler 周期自动执行
```

#### 9.3.4 Phase 2 → Phase 3 的增量升级路径

**Phase 2 架构是 Phase 3 的严格子集**，升级是增量式的：

| 组件 | Phase 2 | Phase 3 | 变化 |
|------|---------|---------|------|
| **接入层** | Channel + FastAPI + CLI | + MCP Server 端点 | **新增** MCP Server 暴露 |

### 9.4 安全架构：从 Policy Engine 到全栈安全

Jarvis 级助手需要更强的安全保障。Phase 3/4 的安全是 Phase 2 Policy Engine 的**扩展**：

| 安全层 | Phase 2 | Phase 3 | Phase 4 |

| **动作门控** | Policy Engine: safe/confirm/blocked | 扩展到 MCP 外部工具 | 扩展到生成代码 |
| **成本控制** | — | 每日 LLM 硬消费上限 + 自动降级 | 同上 |
| **工具审计** | Module 审计日志 | + Tool 调用审计 + MCP 调用审计 | + 生成代码审计 |
| **代码安全** | — | — | AST 扫描 + import 白名单 + 模式检测 + 沙箱 |
| **用户确认** | 高风险操作二次确认 | 外部 MCP 工具首次确认 | 生成代码激活确认 |

**Pincer 安全模型借鉴**（业界最严格的个人 AI 助手安全设计）：
- 用户白名单（只有授权用户能对话）
- 工具审批提示（首次使用新工具需确认）

- 结构化审计日志（所有操作可追溯）
- `pincer doctor` 安全健康检查命令 → OfferPilot 可做 `offerpilot doctor`

## 十、记忆与进化架构：从"无状态工具"到"有记忆的 Jarvis"（深度调研与设计）

> 这是 OfferPilot 成为真正 Jarvis 的最后一块拼图。没有记忆的 AI 只是工具，有记忆的 AI 才是助手。本节是对记忆、人格、偏好学习的**深度调研和完整设计**，确保 Phase 2 的数据管线为未来的记忆与进化能力提供底层支撑。

### 10.1 核心问题：当前架构的记忆缺口

| 能力 | 当前状态 | Jarvis 需要什么？ |
|------|---------|------------------|
| **会话记忆** | ❌ 无。Router/Brain 每次处理消息都是"失忆"状态 | 记住"上次我说过不喜欢 996 的公司"，下次投递自动排除 |
| **事实记忆** | ❌ 无。用户说"我叫张三"，下次就忘了 | 持久记住用户的姓名、偏好、技术栈、求职目标 |
| **操作记忆** | △ 部分。审计日志记录操作，但 Brain 看不到 | 记住"上周投了 20 家公司，5 家回复"，据此调整策略 |
| **人格** | ❌ 无。每次冷启动，没有一致的语气和风格 | 稳定的人格（如：简洁直接、偏技术、不用表情包） |
| **偏好学习** | ❌ 无。用户纠正后不会改变行为 | 用户说"以后别推游戏公司"，永久记住并执行 |
| **反思进化** | ❌ 无 | 从历史交互中自主总结规律，持续变聪明 |

**结论：当前架构的记忆约等于零。这是成为 Jarvis 的最大缺口。**

### 10.2 业界深度调研：AI Agent 记忆系统是怎么做的？

#### 10.2.1 Letta (MemGPT) —— 记忆即操作系统（34K Star）

Letta 的核心创新是**把记忆当操作系统来管理**，而非附加功能：

| 记忆层 | 类比 | 容量 | 特性 |
|--------|------|------|------|
| **Core Memory** | RAM | 有限（<50K 字符，<20 blocks） | 每轮推理都加载，Agent 可自主读写。存放用户关键信息和 Agent 人格 |
| **Recall Memory** | Cache | 中等 | 对话历史的索引存储，按语义或时间检索相关片段 |
| **Archival Memory** | Disk | 无限 | 长期知识库，文档/事实/历史经验。搜索时检索，不常驻上下文 |

**关键设计**：
- **Agent 自主管理记忆**：通过内置工具 `memory_rethink`、`memory_replace`、`memory_insert` 由 LLM 自己决定什么该记、什么该忘
- **记忆即工具**：记忆操作和其他 Tool 一样，是 Brain 推理循环中的 tool_call
- **Context Window 自动管理**：当上下文接近 token 上限时，自动将不重要的信息从 Core 移到 Recall/Archival

**OfferPilot 启示**：三层记忆模型非常适合我们。Core Memory 存用户偏好和 Agent 人格，Recall 存对话历史，Archival 存面经/情报等结构化知识（我们已有 PostgreSQL + ChromaDB 做 Archival）。

#### 10.2.2 Mem0 —— 轻量级记忆层（48K Star）

Mem0 的核心价值是**极简 API + 双存储**：

| 特性 | 说明 |
|------|------|
| **API** | `add(message, user_id)` → `search(query, user_id)` → `update/delete`。3 行代码集成 |
| **双存储** | 向量数据库（语义检索）+ 知识图谱（关系推理） |
| **四类记忆** | Working（当前会话）、Factual（结构化事实）、Episodic（过去经历）、Semantic（通用知识） |
| **自动提取** | 从对话中自动提取值得记忆的事实，无需用户标记 |
| **性能** | 比 OpenAI 原生记忆高 26% 准确率、91% 低延迟、90% 省 token（LoCoMo 基准） |

**OfferPilot 启示**：Mem0 的四类记忆分类和自动提取机制值得借鉴。但 Mem0 的知识图谱功能锁在付费层（$249/月），我们需要自建轻量版。

#### 10.2.3 Zep (Graphiti) —— 时序知识图谱记忆（13K Star）

Zep 的独特之处是**时间作为一等维度**：

| 特性 | 说明 |
|------|------|
| **时序图谱** | 每个事实都有时间戳和有效窗口（"张三在字节工作"→ 2024-2025 有效；2026 后失效） |
| **事实更迭** | 新信息自动覆盖旧信息，保留历史版本（"张三从字节离职了"→ 更新图谱，旧事实标记为过期） |
| **跨会话合成** | 能综合多次对话中的信息碎片，构建完整用户画像 |
| **性能** | DMR 基准 94.8%（超过 MemGPT 93.4%），延迟降低 90%，token 用量 <2% |

**OfferPilot 启示**：时序知识图谱对求职场景特别有价值——"张三 3 月投了字节，4 月收到面试，5 月拿到 offer"这种时间线天然适合图谱表示。但 Zep 依赖 Neo4j/FalkorDB，对个人项目偏重。我们可用 PostgreSQL JSONB + 时间戳实现轻量版。

#### 10.2.4 OpenClaw 记忆与人格系统

OpenClaw 使用**文件系统 + Markdown**管理记忆：

| 文件 | 职责 | 读写权限 |
|------|------|---------|
| `SOUL.md` | 人格定义（身份、语气、价值观、行为边界） | Agent 可读（通常不可写，除非 EvoClaw） |
| `MEMORY.md` | 长期记忆（经验教训、用户偏好） | Agent 可读写 |
| `USER.md` | 用户上下文（名字、角色、习惯） | Agent 可读写 |
| `AGENTS.md` | 行为规则 + 行为记忆（犯过的错不再犯） | Agent 可读写 |
| `memory/YYYY-MM-DD.md` | 每日日志（当天决策和任务记录） | Agent 可读写 |

**EvoClaw 人格进化框架**（157 Star，2026.2）：
- 经验分级：Routine（常规）→ Notable（值得反思）→ Pivotal（重大事件）
- 信念标签：`[CORE]`（不可变：如"我是一个 AI 助手"）vs `[MUTABLE]`（可进化：如"我倾向于用中文回复"）
- 治理级别：Autonomous（自动）/ Supervised（审核）/ Gated（需批准）
- 可视化：Soul 进化时间线 + 信念雷达图

**OfferPilot 启示**：Markdown 文件 + Agent 自修改是一个工程极简但效果显著的方案。EvoClaw 的 CORE/MUTABLE 信念分级和三级治理特别值得采用。

#### 10.2.5 Nova —— 自改进个人 AI（2026.3 发布，DPO 微调）

Nova 是目前唯一在个人 AI 中实现**权重级自改进**的开源项目：

| 能力 | 实现 |
|------|------|
| **纠正学习** | 用户纠正 → 2 阶段检测（regex + LLM）→ 生成 DPO 训练对 {query, chosen, rejected} → 累积 → 自动微调 → A/B 评估 → 部署 |
| **反思层** | 静默检测失败（幻觉、工具循环）→ 生成改进训练对 |
| **好奇心引擎** | 发现知识空白 → 后台异步调研 → 扩充知识 |
| **成功模式库** | 高质量响应存档 → 正强化 |

**硬件要求**：NVIDIA GPU 20GB+ VRAM，本地运行 Qwen3.5:27b

**OfferPilot 启示**：DPO 微调是**终极方案**但硬件门槛高。对于个人项目，我们分两条路：
1. **短期（Phase 3-5）**：无需 GPU 的记忆适应方案（偏好提取 + 规则学习 + Prompt 注入）
2. **长期（Phase 5 进阶）**：预留数据管线和训练对格式，未来可选接入 DPO/RLHF

#### 10.2.6 无需微调的偏好学习方案

| 方案 | 核心思路 | 可行性 |
|------|---------|--------|
| **AgentFly（记忆增强）** | 将历史经验存入 episodic memory，推理时检索相关经验注入上下文 | ✅ 高（不改模型，只改记忆） |
| **Preference Agent（规则蒸馏）** | 从用户纠正中提取自然语言规则，注入 system prompt 引导基础模型 | ✅ 高（我们的 SOUL.md 机制） |
| **In-Context Learning（ICL）** | 将用户偏好表示为 few-shot 示例，每轮推理前注入 | ✅ 高（Core Memory 天然支持） |
| **PersonalAgent（画像构建）** | 对话 → 单轮分解 → 偏好推断 → 用户画像 → 决策引导 | ✅ 中（需要画像提取 Pipeline） |

**关键洞察：不需要 GPU 微调也能实现 80% 的"越用越聪明"效果。** 记忆 + 规则提取 + Prompt 注入就足够。DPO 微调是锦上添花。

#### 10.3.2 Core Memory：Brain 的"工作记忆"

Core Memory 是 Brain 每轮推理时**必定加载**的记忆块，类似人的短期工作记忆：

| Block | 内容 | 来源 | 更新方式 |
|-------|------|------|---------|
| **SOUL** | Agent 人格（语气、风格、行为边界、价值观） | 初始化配置 + 反思进化 | 系统初始化 + 反思 Pipeline（Phase 5）|
| **USER** | 用户画像（姓名、角色、技术栈、求职目标、工作经历） | 对话中自动提取 + 用户手动修改 | Brain 推理中通过 `memory_update` 工具 |
| **PREFS** | 偏好规则（排除条件、通知偏好、优先级） | 用户纠正 + 行为模式推断 | 偏好学习 Pipeline |
| **CONTEXT** | 当前任务上下文（进行中的求职状态、待办事项） | Module 执行后更新 | Module 回调 + Brain 更新 |

**Core Memory 作为 Brain 工具**：

```python
# Brain 可用的记忆工具（与 set_alarm / get_weather 同级）
@tool
async def memory_read(block: str) -> str:
    """Read a memory block. block: SOUL|USER|PREFS|CONTEXT"""
    return core_memory.read(block)

@tool
async def memory_update(block: str, key: str, value: str) -> str:
    """Update a key-value pair in a memory block"""
    core_memory.update(block, key, value)
    return f"Updated {block}.{key}"

@tool
async def memory_search(query: str, limit: int = 5) -> list[dict]:
    """Search recall and archival memory for relevant information"""
    return recall_memory.search(query, limit)
```

**容量管理**：Core Memory 总容量 < 50K 字符（约 12K token）。当接近上限时，Brain 应主动调用 `memory_rethink` 精简 Block 内容（将细节移到 Recall，只保留摘要）。

#### 10.3.3 Recall Memory：对话历史与操作记录

Recall 是所有交互的**全量记录**，Brain 不会全部加载，而是按需检索相关片段：

| 数据类型 | 存储 | 索引方式 | 检索时机 |
|---------|------|---------|---------|
| **对话消息** | PostgreSQL（`conversations` 表） | 时间索引 + 向量索引 | Brain 推理前，自动检索与当前消息相关的历史对话 |
| **工具调用记录** | PostgreSQL（`tool_calls` 表） | 时间索引 + 工具名索引 | Brain 调用 `memory_search` 时 |
| **Module 执行记录** | PostgreSQL（`pipeline_runs` 表） | 时间索引 + Module 名 | Brain 调用 `memory_search` 时 |
| **纠正记录** | PostgreSQL（`corrections` 表） | 时间索引 | 偏好学习 Pipeline 消费 |

**Recall 检索策略**：
1. **时间优先**：默认加载最近 N 轮对话摘要（N 可配，默认 10）
2. **语义检索**：将当前消息向量化，检索语义相似的历史对话片段（Top-5）
3. **合并去重**：两种结果合并去重后注入 Brain 上下文

**与 Phase 2 的关系**：Phase 2 已有的审计日志和 PostgreSQL 数据天然就是 Recall Memory 的数据源。Phase 3 只需增加向量索引和检索接口。

#### 10.3.4 Archival Memory：结构化长期知识

Archival 就是 Phase 2 已有的 **PostgreSQL + ChromaDB** 数据存储。面经库、岗位库、技术情报——这些本身就是 Agent 的长期记忆。

Phase 3 新增的是**时序事实图**（轻量版，不引入 Neo4j）：

```
-- 时序事实表（PostgreSQL）
CREATE TABLE facts (
    id          SERIAL PRIMARY KEY,
    subject     TEXT NOT NULL,           -- "张三"
    predicate   TEXT NOT NULL,           -- "面试了"
    object      TEXT NOT NULL,           -- "字节跳动"
    valid_from  TIMESTAMP NOT NULL,      -- 2026-03-10
    valid_to    TIMESTAMP,               -- NULL = 当前有效
    confidence  FLOAT DEFAULT 1.0,
    source      TEXT,                    -- "对话 session_xxx"
    superseded_by INTEGER REFERENCES facts(id),
    created_at  TIMESTAMP DEFAULT NOW()
);
```

事实更迭示例：
- 用户说"我在准备字节面试" → `(用户, 准备面试, 字节, 2026-03-10, NULL)`
- 后来说"字节面试挂了" → 旧事实 `valid_to = 2026-03-20, superseded_by = new_id`；新事实 `(用户, 面试失败, 字节, 2026-03-20, NULL)`

Brain 通过 `memory_search` 查询时序事实，能回答"我之前面试了哪些公司？结果如何？"

#### 10.3.5 人格机制：SOUL Block + 进化

**SOUL Block 结构**（参考 OpenClaw SOUL.md + EvoClaw）：

```yaml
# Core Memory - SOUL Block (初始化时加载，Agent 可读)
identity:
  name: "OfferPilot"
  role: "个人智能助手"
  purpose: "帮助用户高效完成任务，当前聚焦求职"

style:
  tone: "简洁直接，偏技术"
  language: "中文为主，代码/术语用英文"
  verbosity: "精炼，不废话"
  emoji: false

values:
  - "[CORE] 用户利益优先，不做损害用户的事"
  - "[CORE] 诚实，不确定的事明确说不确定"
  - "[CORE] 高风险操作必须用户确认"
  - "[MUTABLE] 优先推荐远程工作机会"
  - "[MUTABLE] 技术日报控制在5条以内"

boundaries:
  - "不代替用户做最终决定（如接受 offer）"
  - "不发送用户未审核的公开内容"
```

**`[CORE]` vs `[MUTABLE]` 信念分级**：
- `[CORE]`：不可修改的基础信念（安全、诚实、用户利益），即使反思 Pipeline 也不能改
- `[MUTABLE]`：可通过反思和用户反馈进化的偏好/风格

**人格进化流程（Phase 5，借鉴 EvoClaw）**：

```
日常交互 → 经验分级（Routine / Notable / Pivotal）
  ↓
Notable/Pivotal 经验触发反思 Pipeline
  ↓
反思 Pipeline（LLM 分析）：
  "根据最近 30 天的交互，用户似乎更喜欢 {pattern}。
   建议将 SOUL.values 中的 {rule} 更新为 {new_rule}。"
  ↓
治理门控：
  - MUTABLE 信念 + Supervised 模式 → 推送给用户审核
  - MUTABLE 信念 + Autonomous 模式 → 自动应用（低风险偏好调整）
  - CORE 信念 → 永远不自动修改
  ↓
应用更新 → 记录变更日志（可追溯、可回滚）
```

#### 10.3.6 偏好学习：无需 GPU 的"越用越聪明"

**Track A：规则提取式偏好学习**（Phase 3-5，不需要 GPU）

| 学习信号 | 检测方式 | 转化为 |
|---------|---------|--------|
| **用户纠正** | 用户说"以后别推游戏公司" | PREFS 规则：`exclude_company_tags: ["游戏"]` |
| **用户反馈** | 日报标记"无用" | 降低该来源/标签的权重 |
| **隐式行为** | 用户每次都忽略 XX 类推送 | 推断偏好 → 推送给用户确认 → 写入 PREFS |
| **工具偏好** | 用户总是手动修改 LLM 的投递打招呼语 | 提取用户偏好的打招呼风格 → 更新 PREFS |

纠正检测流程：

```
用户消息 → 纠正检测器（regex 预筛 + LLM 判定）
  ↓ 是纠正
提取规则（LLM）："用户不希望收到游戏公司的推荐"
  ↓
写入 Core Memory PREFS Block
  ↓
同时生成 DPO 训练对（Phase 5 进阶备用）：
  {query: "推荐岗位", rejected: "XX游戏公司...", chosen: "XX科技公司..."}
  ↓
存入 training_pairs 表（仅收集，不训练）
```

**Track B：DPO 权重微调**（Phase 5 可选，需要 GPU，预留数据管线）

Phase 3 开始收集训练数据，Phase 5 有 GPU 时可选接入训练管线：

| 数据源 | 训练对格式 | 积累时机 |
|--------|-----------|---------|
| 用户纠正 | `{query, chosen(用户修改版), rejected(原始版)}` | 实时 |
| 反思结论 | `{query, chosen(优化策略), rejected(旧策略)}` | 反思 Pipeline |
| 成功模式 | `{query, chosen(高评分响应), rejected(低评分响应)}` | 用户反馈 |

**为什么先不训练**：
1. 个人助手的交互量有限，需要几个月积累足够训练对
2. DPO 微调需要 GPU（20GB+ VRAM），不是所有用户都有
3. 规则提取 + Prompt 注入已能实现 80% 的个性化效果
4. 数据先攒起来，格式对了，未来一键接入训练管线

#### 10.3.7 记忆在 Brain 推理循环中的集成

更新后的 Brain 推理循环：

```python
async def brain_loop(message: str, context: BrainContext) -> str:
    # === 记忆加载阶段 ===
    core = core_memory.load_all()                     # Layer 1: 加载 SOUL + USER + PREFS + CONTEXT
    recent = recall_memory.recent(n=10)               # Layer 2: 最近 10 轮对话摘要
    relevant = recall_memory.search(message, top_k=5) # Layer 2: 语义检索相关历史
    
    system_prompt = build_system_prompt(
        soul=core["SOUL"],
        user=core["USER"],
        prefs=core["PREFS"],
        task_context=core["CONTEXT"],
        recall=merge_deduplicate(recent, relevant),
    )
    
    tools = (
        tool_registry.list_tools()       # Ring 1: Built-in Tools
        + module_registry.as_tools()     # Ring 2: Modules
        + mcp_client.list_tools()        # Ring 3: MCP
        + memory_tools()                 # 记忆工具: memory_read/update/search
    )
    
    context.set_system(system_prompt)
    context.append({"role": "user", "content": message})
    
    # === ReAct 推理循环 ===
    for step in range(MAX_TOOL_CALLS):
        response = await llm_router.call(
            messages=context.messages,
            tools=tools,
        )
        
        if response.is_text():
            break
        
        for tool_call in response.tool_calls:
            result = await execute_tool(tool_call, context)
            context.append(tool_call_result(tool_call, result))
    
    # === 记忆写入阶段 ===
    recall_memory.save_turn(message, response.text, context.tool_calls)
    correction_detector.check(message, context)  # 检测是否包含纠正信号
    
    return response.text
```

| **自主管理** | Agent 读写 Markdown 文件 | 无 | 自动 | Agent 通过工具编辑 Memory Block | Agent 通过 `memory_*` 工具管理 |

**OfferPilot 的独特优势**：
1. **四层记忆** 比大多数竞品多一层 Meta Memory（进化层），是"越用越聪明"的结构基础
2. **双轨学习** 既有无需 GPU 的规则提取（实用），又预留了 DPO 微调（终极）
3. **信念分级** 借鉴 EvoClaw 的 CORE/MUTABLE 标签，确保安全底线不被进化突破
4. **增量兼容** Phase 2 的数据管线（PG + ChromaDB + 审计日志）天然就是 Archival + Recall 的数据源

### 10.5 为什么不直接使用 Mem0 / Letta / Zep？

| 方案 | 不采用的原因 | 我们借鉴了什么 |
|------|------------|-------------|
| **Mem0** | 知识图谱锁在付费层；是 SaaS 优先（自托管是附属）；引入重依赖 | 四类记忆分类、自动提取机制、简洁 API 设计 |
| **Letta (MemGPT)** | 重框架（自有 Agent Runtime）；架构假设与我们不同（我们已有 Brain） | 三层记忆模型、Agent 自主管理记忆的理念、容量管理策略 |
| **Zep (Graphiti)** | 依赖 Neo4j/FalkorDB（个人项目过重）；SaaS 优先 | 时序事实追踪、事实更迭机制 |
| **Nova** | DPO 微调需 GPU 20GB+；AGPL 许可证限制 | DPO 训练对格式、纠正检测机制、双阶段学习 |

**我们的策略：不引入重框架，自建轻量版，cherry-pick 各方案精华**。核心代码量预估 < 2000 行（`core/memory/` + `core/soul/` + `core/learning/`），与 Pincer 全量 7,800 行的"小而美"理念一致。
